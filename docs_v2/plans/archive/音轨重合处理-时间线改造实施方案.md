# 音轨重合处理（NLE 规则）& 时间线改造实施方案

日期：2026-02-06  
适用页面：`/dashboard/projects/video-editor/[id]`（别名）与现有编辑器页面  
目标：把“字幕语音轨道”的重合问题一次性做对：**同轨不重叠、交互直观、实现足够稳**，并引入 **wavesurfer.js + wavesurfer-multitrack** 做波形与多轨预览（可渐进上线）。

硬约束：**不调整任何外部服务（Python/Java）实现**；只在 Next.js / DB / R2（已存在 API 能力）范围内落地。

---

## 0. 已确认的决策（按推荐方案执行）

- 交互规则：**同轨不允许重叠**（默认吸附卡住）。
- 高级操作：**支持 Shift Ripple**（仅在用户按住 Shift 且发生冲突时推开后续；默认不自动移动其他片段）。
- 合成链路：外部合成服务保持 legacy `audio_clips: string[]`（依赖 **id 编码时间**）；时间线保存时通过 **id 重命名 + R2 覆盖复制** 保证“改时间线 = 改输出”。
- 波形方案：**采用 wavesurfer.js**（先做“长波形背景参考”），multitrack 第二阶段再上。

## 1. 结论（先定死规则，避免反复返工）

### 1.1 行业主流（推荐采用）
采用 **NLE（剪辑软件）规则**：
- **同一条轨道不允许重叠**（Voice/BGM/… 都一样）。
- 需要“叠加播放”的，必须放到**另一条轨道**（未来再扩展多条 Voice 轨也行，但第一版先不做）。

### 1.2 我们产品的默认交互（KISS）
默认行为（不需要用户理解“模式”）：
- 拖动片段遇到邻居片段时：**吸附 / 卡住**（magnet），不允许落到重叠区。
- 同轨片段总是保持按时间顺序排列（稳定性 > 花活）。

高级操作（已确定实现，但不污染 UI）：
- **按住 Shift = Ripple（推开后续）**：当用户拖动导致冲突，按住 Shift 则将后续片段整体右移，消除冲突。
  - UI 表达：只在发生冲突时浮出一次轻提示 `Shift 推开后续`（1.2s 自动消失，不常驻按钮）。

非目标（第一版不要做）：
- 同轨混音、同轨交叉淡入淡出编辑、包络线、自动闪避（ducking）面板、复杂 DAW 玩法。

## 2. “重合”的根因拆解（别只修 UI）

字幕语音“重合”通常来自两类问题：
1) **时间窗重合**：字幕段（start/end）本身相互覆盖（SRT 有时会出现）。
2) **音频超窗**：某段 TTS 音频实际时长 > 字幕段分配时长（end-start），导致合成时覆盖下一段。

要“稳定可用”，必须做到：
- 前端编辑时：**永远不能生成同轨重合的时间窗**。
- 后端合成时（外部服务暂不改）：短期先保证“时间窗不重叠 + 交互稳定”，并在 UI 做**超窗提示**；真正的 trim/tempo/fade 兜底留到外部合成服务允许改造时再补齐。

## 3. 数据与接口改造（决定你能不能真的改时间线）

### 3.1 现状（Repo 真实落点，方便排期）
当前链路（需要改造的关键节点）：
- 触发合成：`src/app/api/video-task/generate-video/route.ts`
- 调 Python：`src/shared/services/pythonService.ts` → `pyMergeVideo(taskId, nameArray)`
- Python 目前入参：`audio_clips: string[]`（高概率靠 `id` 编码解析时间）
- 字幕 JSON 存储：`vt_task_subtitle.subtitleData`（jsonb）
  - 单条保存：`src/app/api/video-task/update-subtitle-item/route.ts`（内部用 `updateSingleSubtitleItem`）
  - SQL 更新：`src/shared/models/vt_task_subtitle.ts` → `updateSingleSubtitleItem`

> 结论：长期看“显式时间（结构化）”才是正道；但本次硬约束 **不改外部服务**，所以先用 **id 重命名链路** 保证“改时间线 = 改输出”，并把显式时间作为后续可选升级项。

### 3.2 理想方案（需要外部服务支持，暂不实施）：合成协议改为显式时间
目标：让 id 成为稳定主键，不再承载时间语义。

改造点：
- `POST /api/video-task/generate-video`
  - 当前：`audio_clips: string[]`
  - 目标：`audio_clips: Array<{ id: string; startMs: number; endMs: number; path: string }>`
    - `path` 可以用相对路径（例如 `adj_audio_time/<id>.wav`），由 Python 拼 bucket 前缀。
    - start/end 用毫秒（避免字符串解析误差）。
- Python `/api/internal/audios/video/merge`
  - 兼容策略（必须做）：如果收到对象数组就用对象的 start/end；如果还是 string[] 就按旧逻辑解析（兼容老任务/旧前端/旧服务）。

好处：
- 前端拖时间线 = 直接改 start/end（不需要改 id，不需要搬文件）。
- 后端合成逻辑从“字符串解析”升级为“结构化数据”，少坑。

### 3.3 本次落地方案（推荐）：维持“id 编码时间”，实现重命名链路
在 **不调整外部合成服务协议** 的前提下，唯一能让“时间线编辑影响最终输出”的方案：
- 拖动导致 start/end 改变时（批量保存接口一次性处理）：
  1) 生成新 id（同 seq 前缀，时间段更新）：`0001_<start>_<end>`
  2) 更新 subtitleData：把该 item 的 `id/start/end` 一起更新
  3) R2 内把音频文件**覆盖复制**：`adj_audio_time/<oldId>.wav` → `adj_audio_time/<newId>.wav`
  4) 返回 `idMap(old->new)` 给前端，同步本地状态（否则下一次保存会找不到旧 id）
- 约束/策略：
  - 不做删除旧文件（避免失败回滚困难；后续可用离线清理）
  - Java 覆盖复制需要并发上限（例如 4）防止雪崩
  - 只对符合 legacy 格式的 id 做重命名（否则保持原 id，只更新 start/end）

## 4. 前端时间线编辑（核心交互 + 稳定性策略）

### 4.1 需要的最小交互（别上来就“专业剪辑”）
在当前编辑器的字幕轨道上新增（现实现落点：`src/app/[locale]/(dashboard)/video_convert/video-editor/[id]/timeline-panel.tsx` + `src/shared/components/video-editor/subtitle-track.tsx`）：
- 片段 **拖动**：改变 `startTime`（保持 duration 不变）。
- 可选：左右边缘 **调整**（是否允许改 duration 取决于产品规则；第一版建议不开放，避免和音频实际时长冲突）。

提示与反馈（极简但明确）：
- **吸附**：接近邻居边缘时自动贴边（视觉：轻微“卡住”+ 边缘高亮 150ms）。
- **冲突**：用户强行拖入重叠区时，片段回弹并短暂红描边（200ms），同时在冲突点出现 1px 红线。
- **高级提示**：只有在冲突发生时，浮出 `Shift 推开后续`（1.2s 后消失；按住 Shift 则直接 Ripple）。

实现方式（优先 KISS，不引入不必要复杂度）：
- 拖拽交互优先用 **pointer events + CSS transform 预览**（拖动过程不频繁 setState 全量列表，避免卡顿）。
- 仅在 `pointerup`（落点确认）时：
  1) 套用“同轨不重叠算法”得到最终 startTime；
  2) 更新 `subtitleTrack`；
  3) 标记 dirtyMap。

### 4.2 同轨不重叠算法（纯函数，先写对再接 UI）
建立一个无副作用的策略模块（便于测试）：
- 输入：当前轨道 clips（按 startTime 排序）、被操作 clipId、操作类型（move/resizeStart/resizeEnd）、候选新时间、是否 ripple
- 输出：新的 clips（保证非重叠、在 [0,total] 内）

Overwrite（默认吸附/卡住）规则（伪代码）：
```
prevEnd = end(prevClip)
nextStart = start(nextClip)
newStart = clamp(candidateStart, prevEnd, nextStart - clip.duration)
```

Ripple（Shift 推开后续）规则（只在冲突时触发）：
```
delta = candidateStart - clip.start
clip.start += delta
for each nextClip after clip:
  if nextClip.start < clip.end:
    push = clip.end - nextClip.start
    nextClip.start += push
    clip = nextClip
```

注意：
- 只处理“同轨”即可；跨轨不需要碰撞检测（第一版）。
- 每次操作完成后重新 sort（保证稳定）。

建议落点（便于测试/复用）：
- 纯函数：`src/shared/lib/timeline/collision.ts`
- 单测：`src/shared/lib/timeline/collision.test.ts`（vitest）

### 4.3 状态与保存策略（避免拖一次就打 20 个 API）
拖动时间线会产生“批量连锁修改”（Ripple 时尤其明显）。必须：
- 前端只维护一份权威状态：`subtitleTrack`（按时间排序）。
- 维护一个 `dirtyMap`：记录被修改过的条目（id -> {startMs,endMs}）。
- 保存策略：
  - 默认：用户点击“合成视频”前自动检查 dirtyMap，**先一次性保存**再触发合成。
  - 额外：提供一个不抢眼的 `已修改 · 待保存` 计数（已有 pendingChanges 模式可复用）。

后端新增（推荐）：
- `POST /api/video-task/update-subtitle-timings`（批量）
  - payload 只带必要字段：`taskId`, `stepName='translate_srt'`, `items:[{id, start, end}]` 或 `{seq,start,end}`
  - SQL：单次更新整段 jsonb（或按 seq 替换），不要循环 200 次 update。

Repo 落点建议：
- API：`src/app/api/video-task/update-subtitle-timings/route.ts`
- Model：`src/shared/models/vt_task_subtitle.ts` 新增批量更新方法（一次 SQL）

## 5. 引入开源：wavesurfer.js + multitrack（渐进式，不要“一口吃成胖子”）

### 5.1 选型理由（稳定优先）
- `wavesurfer.js`：维护活跃（版本迭代到 2025-12 仍在更新），许可证 BSD-3-Clause，生态成熟。
- `wavesurfer-multitrack`：同作者系生态组件，可用来做多轨波形同步。

### 5.2 第一阶段的最小集成（只做“看得懂”，不做“全 DAW”）
目标：让用户在时间线上“更有把握”，不靠猜。

做法：
- 只渲染 **1~2 条长波形**（而不是每个字幕片段一条波形）：
  - 原始音频（参考）
  - BGM（可选）
- 字幕语音仍用“块状片段”表达（清晰、轻量），波形只做背景参考。

实现要点：
- wavesurfer 组件必须 `dynamic import`（仅 client），避免 SSR 崩。
- **CORS 兜底**：私桶签名 URL 在 WebAudio 解码时可能遇到 CORS/`Failed to fetch`，不改外部服务的前提下，用 Next 增加一个同源代理（`/api/storage/proxy?src=...`）来转发音频流，保证 wavesurfer 稳定加载。
- 波形渲染与时间线滚动宽度同步（用 `ResizeObserver` 监听容器宽度）。
- 只在编辑器区域可见时初始化，离开页面销毁（避免内存泄露）。

### 5.3 第二阶段（可选）：多轨波形同步
当第一阶段稳定后，再考虑 multitrack：
- 同步 cursor / seek 与现有 video currentTime。
- 只做“预览同步”，不把播放引擎完全替换（不要一次性重写播放系统）。

## 6. 限制：音频超窗（外部合成服务暂不可改）

无论前端多完美，只要某段 TTS 音频真实时长 > 分配时间窗（end-start），最终合成仍可能压到下一句。

在“**不改外部服务**”约束下，本次可落地的处理方式只有两类：
1) **UI 提示 + 引导修正**
   - 标记超窗段（例如：窗内 2.0s / 音频 2.6s）
   - 给出建议操作：`延长当前段`（若后方有空隙）或 `Shift 推开后续`（ripple）
2) **可选的数据补齐（Next 侧可做）**
   - 在生成/保存音频时把 `durationMs` 写回 subtitleData，供 UI 判断与提示

真正的 trim/tempo/fade 兜底只能在外部合成服务允许改造时完成（列为后续升级项）。

## 7. 轨道多/接口多导致加载慢：更友好的处理策略（必须做）

问题本质：**“每条轨道/每段音频单独打接口”** 会带来两类灾难：
- 网络层：并发过高 → 排队、超时、失败重试雪崩；
- 交互层：首屏一直转圈 → 用户感知“卡/不可靠”，尤其在编辑场景会直接放弃。

目标：让用户在 1~2 秒内进入可编辑状态，剩余内容后台补齐。

### 7.1 先从“少打接口”开始（最有效，最省代码）
1) **Bootstrap 聚合接口（一次拿齐基础数据）**
   - 把“编辑器首屏所需”合并到一个接口里返回：任务状态、视频 URL、BGM URL、字幕数组（start/end/text）、必要的 r2 前缀与 env。
   - 对“重资源”（波形 peaks / 音频真实时长统计）用 `include=` 参数按需取，不默认返回。
2) **批量接口替代 N 次单条接口**
   - peaks/metadata 这类：提供 `POST /api/.../batch`，入参一组 path/id，返回 `Record<id, data>`。
3) **避免前端“为了一点信息”拉整段大 JSON**
   - 例如：右侧列表不需要一次性拿所有音频文件；只要 URL 规则 + 需要时播放再加载即可（当前做法是对的）。

### 7.2 分级加载：先可用，后完整（用户不等）
建议拆成三层：
- L0（立即可用，<= 1~2s）：页面框架 + 视频预览 + 时间刻度 + 字幕块（无波形）。
- L1（快速补齐，后台加载）：可见区域的波形 / 统计信息（只加载 viewport 内的轨道）。
- L2（可选、空闲再做）：其余轨道波形、较高精度 peaks、预取下一屏。

实现建议：
- 轨道容器用 `IntersectionObserver`：只有轨道进入可视区才触发加载。
- 用 `requestIdleCallback`（或退化为 setTimeout）做“空闲预取”，避免抢 UI 主线程。

### 7.3 并发与取消（防止“越加载越慢”）
最小但必须的工程策略：
- **并发上限**：peaks/波形请求最多 3~4 个并行（其余排队）。
- **可取消**：页面切换/折叠轨道时，用 `AbortController` 取消未完成请求。
- **失败退避**：波形加载失败不要阻塞编辑；显示“波形稍后重试”并在后台指数退避重试 1~2 次即可。

### 7.4 缓存（让第二次打开“秒开”）
优先级从低成本到高收益：
1) 内存缓存（本次会话内重复打开不再拉）。
2) HTTP 缓存（ETag/Cache-Control），peaks 这类静态数据可长缓存。
3) 持久缓存（localStorage/IndexedDB）：
   - key 用 `audioPath + etag/hash`，音频变了自动失效。
   - 只存 peaks（小 JSON），不要存音频本体。

### 7.5 UI 反馈（不焦虑、不打扰，但让人“放心在跑”）
UI 不需要花哨，但必须“可信”：
- 顶部只显示一个极简状态：`已就绪 · 波形加载中 3/12`（可点击展开查看失败项）。
- 轨道本身用“空壳波形”占位：细灰线 + 轻 shimmer，不让布局跳动。
- 不要全屏 loading 覆盖编辑器（除非 L0 基础数据都没回来）。

> 关键原则：**编辑不依赖波形**。波形是增强信息，不应该成为“阻塞点”。

## 8. 验收标准（写清楚，不要靠感觉）

时间线编辑：
- 同一轨道任意时刻 `end(i) <= start(i+1)` 恒成立（保存前后都成立）。
- 拖动/吸附手感稳定：长列表（>300 段）仍能保持流畅（不出现明显卡顿）。
- Ripple 仅在用户显式按 Shift 时触发；默认不会“偷偷移动其他片段”。

合成一致性：
- 前端时间线看到的 start/end 与最终生成视频的对齐一致（抽样验证 10 段以上）。

性能与稳定性：
- wavesurfer 初始化/销毁无泄漏；切换页面不累积 AudioContext。
- 批量保存接口单次完成（不出现 N 次请求风暴）。
- 轨道很多时（例如 12+）首屏仍能在 1~2 秒进入可编辑状态；波形/统计后台补齐，不阻塞。

## 9. 实施里程碑（按风险从低到高分阶段交付）

### M0（1 天）：Legacy id 重命名链路（先保证“改时间线 = 改输出”）
- Next 侧新增批量保存：更新 subtitleData 的 `start/end/id`，并通过 Java 覆盖复制把 `adj_audio_time/<oldId>.wav` 写到 `adj_audio_time/<newId>.wav`
- 返回 `idMap(old->new)`，前端同步本地状态（避免下一次保存找不到旧 id）
- `generate-video` 继续只发送 legacy `audio_clips: string[]`（不调整外部服务）

### M1（2~3 天）：前端“同轨不重叠”+ 交互落地（不引 wavesurfer）
- 字幕轨实现拖动（pointer events；Shift Ripple）。
- 引入纯函数策略模块 + 单元测试（重合、边界、Ripple）。
- dirtyMap + “合成前自动保存”策略（接口可先 mock，但 UI 行为要先稳定）。

### M2（1~2 天）：后端批量保存接口 + 合成对齐
- 增加 `update-subtitle-timings` 批量接口（含重命名 + R2 覆盖复制）。
- “合成视频”前确保 subtitleData 已保存成功（失败则阻断合成，并给出明确提示）。

### M3（2~4 天）：wavesurfer 第一阶段（长波形背景）
- 只做原始音频波形（+ 可选 BGM）。
- 与时间线宽度/缩放/滚动同步。

### M4（1~2 天）：音频超窗提示（外部服务不改的前提下）
- 记录并展示每段音频 `durationMs`（若能获取），并在 UI 标记“超窗”
- 提供引导操作：延长当前段（若有空隙）/ Shift Ripple 推开后续

## 10. 说明（为什么这套更稳）
- 规则先定死：同轨不重叠 + Shift Ripple，避免“拖一拖就乱”。
- 短期不改外部服务：用 **id 重命名 + R2 覆盖复制** 让旧合成链路继续工作；长期再升级显式时间协议。
- wavesurfer 渐进：先提升“可理解性”，再考虑“专业性”，避免一次性大重构导致不稳定。
