---

## 0) 产品一句话

SoulDub/ReVoice：用户上传视频 → 自动生成“目标语言配音 + 字幕”的新视频，并提供字幕编辑、
单段音频重生成、重新合成、下载成品等能力。

---

## 1) 核心用户流程（你提需求必须对齐）

1) 登录/注册（邮箱验证码 / Google / GitHub 等，取决于后台配置）
2) 上传视频（分片直传 R2）
3) 创建翻译任务（写 DB + 扣积分）
4) 轮询进度（DB 为真相源；可选读取 Java 的 steps 聚合）
5) 产物就绪后下载（只信 DB 的“成品清单”）
6) 字幕编辑闭环：
   - 单句翻译/改文案
   - 单段音频重生成（TTS）
   - 合成新视频（job 化，支持刷新可恢复）

端到端链路与职责边界：`04-architecture.md`

## 2) 现状能力地图（你想改哪里）

建议先读：`03-feature-map.md`

## 3) 关键“不可变约束”（不对齐会导致返工）

- **DB 是真相源**：
  - “任务是否完成/文件是否可下载”以 DB 表为准，不允许用前端去探测 R2 是否存在文件。
- **id 体系**：
  - `fileId`/`taskId` 是端到端链路的关键锚点（影响 R2 路径、幂等、恢复）。
- **多系统协同**：
  - Web（本仓库）改动，经常需要同时对齐 Java（video-tools）与 Python（VAP/TTS）。

## 4) 你提需求时，最有效的信息（建议直接按模板写）

需求模板与验收清单：`09-requirements-iteration.md`

最关键的 6 件事：

1) 用户是谁（免费/订阅/管理员？）
2) 目标是什么（流程中的哪一步要变？）
3) 输入/输出是什么（数据/文件/状态机/计费规则）
4) 是否影响积分/订阅/支付（价格、扣费点、退款、优先级）
5) 是否影响外部依赖（Java/Python/R2/Stripe 等）
6) 验收怎么测（最小可重复步骤 + 预期结果）
