# 需求迭代流程（模板 + 评审清单 + 验收清单）

> 最后校验：2026-03-04  
> 目标：让需求“描述清楚、影响面收敛、研发能快速落地、可验收可回归”。

---

## 1) 需求迭代的最小闭环

1) 需求提出（按模板）
2) 影响面评审（Web / Java / Python / DB / R2 / 支付）
3) 方案落地（开发 + 测试）
4) 验收（按清单）
5) 文档回写（本目录是新单一入口）

---

## 2) 需求模板（建议复制后填写）

```text
【背景/问题】
现状是什么？卡在哪里？影响谁？

【目标】
你要达成的用户价值是什么？成功标准是什么？

【用户与权限】
涉及哪些角色：未登录 / 登录用户 / 订阅用户 / 管理员？

【流程位置】
对应核心链路哪一步：登录 / 上传 / 创建任务 / 进度 / 下载 / 字幕编辑 / 支付？

【输入/输出契约】
输入字段/文件是什么？输出字段/文件是什么？是否变更数据结构？

【计费与权益】
是否影响积分扣费、订阅赠送、免费额度、优先级？

【外部依赖】
是否需要改 Java video-tools？是否需要改 Python VAP/TTS/Speaker？
是否涉及 Stripe/PayPal/Creem 等支付平台？

【埋点/可观测】
是否需要新增日志、指标、告警？

【验收步骤】
给出最小可复现步骤（含测试账号/测试数据） + 预期结果
```

## 3) 影响面评审清单（研发/产品一起过一遍）

### 3.1 Web（本仓库）

- UI 入口是否在 `src/app/[locale]/(dashboard)` / `(admin)` / `(landing)`？
- 是否需要新增/调整 BFF API（`src/app/api/**`）？
- 是否涉及 DB schema（`src/config/db/schema.ts`）或 models（`src/shared/models/*`）？

### 3.2 Java（video-tools）

- 是否涉及 R2 预签名/multipart？
- 是否涉及调度策略/任务状态写入/成品清单（`vt_file_final`）？
- 是否涉及 internal 回调与鉴权？

### 3.3 Python（VAP/TTS/Speaker）

- 运行形态是 Modal Job 还是 FastAPI？（见 `04-architecture.md`）
- 并发与超时门禁在哪里调整？
- 是否需要新产物写入 R2？对应 DB 清单怎么落？

### 3.4 安全与合规

- 是否新增公开端点？是否会被第三方回调调用？
- 是否涉及代理用户提供的 URL？（SSRFi 风险）
- 是否涉及隐私/版权/支付条款变更？（需要法务评审）

---

## 4) 验收清单（建议每个需求都能跑）

- 功能：按需求验收步骤走通
- 权限：未登录/普通用户/管理员边界正确
- 计费：扣费点正确；失败是否退款/回滚；幂等是否成立
- 性能：大文件上传/长任务不会因为超时导致不可恢复
- 可观测：关键错误有日志；必要时有告警指标
- 回归：`pnpm test`、`pnpm test:db`、涉及上传则跑 `pnpm test:multipart`
