# SoulDub定价策略

# 原则

- 登录前允许上传视频
- 登录后才可生成翻译视频
- 未登录用户在点击生成时，引导用户到注册/登录页面
- 视频翻译每分钟消耗3积分（计费分钟 = `ceil(durationSeconds / 60)` 向上取整，且最少按 1 分钟计费）
- 订阅策略要简单明了，不要让用户有选择困难
- 订阅分为免费版、标准版、高级版、企业版(没啥用，但是个噱头，看上去专业可信)
- 包月和包年用户权益一致，包年用户可免两个月订阅费
- 单独买积分要比包月和包年略贵，否则用户宁愿单独买积分，而不是包月和包年了

# **订阅周期与积分规则**

## 1 订阅周期与扣款

- 订阅按所选周期自动续费：**包月**或**包年**。
- 每期的 **开始/结束/下次扣款时间** 优先以 PayPal/Stripe 的 API 返回为准；若某平台/场景未返回到期时间，再按「对日规则」基于 interval 做兜底计算。

## 2 订阅权益生效与失效

- 订阅权益以“该期有效期”为边界：在本期内有效，到期即失效（除非续费成功进入下一期）。
- 若用户取消订阅：

取消=关闭自动续费，**不等于退款**。

取消后：当前已支付周期内订阅权益/订阅积分发放规则不变，**有效至到期日**。

到期后：不再自动扣款；订阅积分不再发放；账户可继续使用**未到期的充值积分，以及继续享有免费用户权益。**

## 3 订阅积分

### 3.1 发放规则

- **月付**：每个订阅期（每月）发放一次订阅积分。
- **年付（按月发放）**：年付有效期内，仍按“每个订阅月”发放一次订阅积分（即每月一发）。

### 3.2 有效期与结转

- 订阅积分**仅在当期内有效**：到该订阅月期末即过期。
- 订阅积分**不结转**：当期未用完自动失效。

## 4 充值积分

### 4.1 批次与有效期

- 每次充值生成一个独立“批次”。
- 每个批次自充值成功起 **有效期12个月**，到期该批次剩余积分失效。
- 多批次同时存在时，各自独立到期，互不影响。

### 4.2 扣减顺序

- 消费时积分扣减顺序的原则：**谁先到期先用谁**

1. **全局统一按 expiresAt 排序**：`expiresAt` 越早越先扣；`expiresAt=NULL`（永不过期/历史兜底）最后扣。
2. 多批次充值积分内部也采用**先到期先用**：优先消耗最早到期的充值批次。

### 4.3 生效开始时间（availableAt）

> 结论：不需要“pending”状态。积分是否可用完全由时间边界推导。

- 每个积分批次有两条时间边界：
  - `availableAt`：可用开始时间；`NULL` 表示“立即可用”
  - `expiresAt`：到期时间；到期后剩余积分失效
- 可用性判定（服务端口径）：
  - `now < availableAt`：未生效（不计入可用余额；不参与扣减顺序）
  - `availableAt <= now < expiresAt`（或 `expiresAt=NULL`）：可用
  - `expiresAt <= now`：已过期（不计入可用余额）
- “限时特惠（$5.99/3 天）”落地口径：
  - 每次购买产生 1 个积分批次，`availableAt=该次 3 天窗口开始`，`expiresAt=该次 3 天窗口结束`
  - 有效期内复购时，新批次 `availableAt` 对齐到“当前到期日”，实现“下一段 3 天开始才可用”，且每段窗口只生效 1 笔（不叠加）。

## 5 订阅与充值积分叠加

- 用户同时拥有订阅积分与充值积分时：
- 积分可叠加使用；
- 消耗遵循“先到期先用”；
- 订阅积分仍按月发放与按月过期，不因充值而改变。

## 6 状态变化对积分的影响

- **续费成功进入下一期**：按新期规则发放当期订阅积分；上一期剩余订阅积分不结转。
- **到期未续费**：订阅积分停止发放；已发放但未用完的订阅积分在期末失效；未到期的充值积分仍可使用直至各批次到期。
- **退款**：仅提供未使用的积分充值费用退款，退款后回收对应的充值积分；订阅费用不予退款。

## 7 对日规则

- 包月

每月同一日期同一时间扣费；若下月无该日期，则为下月最后一天同一时间。

1 月 31 日开通 → 2 月没有 31 日，到期日就落到 2 月最后一天（2月28/29 日，根据是否闰年）

8 月 31 日开通 → 9 月没有 31 日，到期日就落到 9 月最后一天（9月30日）

- 包年

每年同一月同一日同一时间扣费；若次年无该日期（如2/29），则为该月最后一天同一时间。

2 月 29 日（闰年）开通 → 下一年是非闰年，到期日就落到 2 月 28 日

## 8 订阅升级/降级（标准↔高级）口径

- 下一账期月生效。

## 9 免费和赠送

- 取消免费权益
- 取消新用户一次性赠送权益
- 增加限时特惠权益

## 10 同套餐重复订阅（包月 / 包年）

V1 口径（KISS）：**不支持“重复订阅叠加顺延”**。

- 同一用户同一时间仅允许存在 **1 条有效订阅**（以支付平台 subscriptionId 为准）。
- 若用户在有效期内再次下单订阅（同套餐/跨套餐），统一引导到 Billing Portal 做升级/降级/续费管理（避免双订阅导致权益重复、对账困难）。

## 11 跨套餐重复订阅（包月↔包年）

同上：V1 不支持叠加/顺延，统一走 Billing Portal 的“变更订阅”能力。

## 12 自动续费规则

1. 扣费绑定：仅和**最后一次订阅的套餐类型**有关；
2. 扣费时间：**到期日的同一具体时间点**自动扣款（如到期 3 月 10 日 10:00，就 3 月 10 日 10:00 扣款）；
3. 扣款失败处理：补扣（retry/dunning）优先依赖 PayPal/Stripe 平台侧能力；当前版本不在应用侧额外实现补扣。
4. 顺延规则：扣款 / 补扣成功后，立刻按绑定套餐类型**对日对时**顺延周期（如 3 月 10 日 10:00 扣款成功，包月顺延至 4 月 10 日 10:00）。

> 说明：若未来需要在应用侧实现“补扣次数/时间窗口/失败降级”等策略，再单独立项做风控与财务口径对齐。

> 目标：把“文档口径 / 价格表展示 / 支付结账 / 赠送积分 / 扣积分 / 权益限制”统一到一套**可落地且可审计**的规则上，避免前后端各算各的、以及安全/刷积分漏洞导致定价失效。

## 1. 当前代码里的真实逻辑（实际生效的 Source of Truth）

### 1.1 定价数据从哪里来

- 定价、credits、valid_days 等**业务数值**来自 i18n JSON（不是数据库/CMS）：
  - `src/config/locale/messages/zh/pricing.json`
  - `src/config/locale/messages/en/pricing.json`
- 结账接口按 `locale` 读取定价 JSON，并用其中的 `amount/currency/interval/payment_product_id/credits/valid_days` 创建订单：
  - `src/app/api/payment/checkout/route.ts:30`（读取翻译）
  - `src/app/api/payment/checkout/route.ts:36`（按 `product_id` 找 item）
  - `src/app/api/payment/checkout/route.ts:103`（金额/币种服务端重新计算，不信任前端传入）
  - `src/app/api/payment/checkout/route.ts:132`（interval 校验）
  - `src/app/api/payment/checkout/route.ts:256`（把 credits/valid_days 写入订单）

### 1.2 支付成功如何发放积分（订阅/充值）

- 支付成功后会基于订单里的 `creditsAmount/creditsValidDays` 创建一条 `credit(grant)`：
  - 首次订阅/首次支付：`src/shared/services/payment.ts:111-229`
  - 续费：`src/shared/services/payment.ts:371-488`
- 积分过期策略：
  - one-time（充值积分）：按批次 12 个月有效；到期时间按“充值成功起 12 个月”（对日，无该日期取月末）计算。
  - one-time（其它/兜底）：`creditsValidDays <= 0` => 永不过期（`expiresAt = null`）
  - 订阅：以 `subscription.currentPeriodEnd` 作为到期时间（与 `creditsValidDays` 无关）
  - 见：`src/shared/models/credit.ts:33-75`

### 1.3 视频翻译怎么扣积分（关键：后端的成本规则）

- 创建视频翻译任务时扣积分，成本来自系统配置 `credit.points_per_minute`（默认回退 3）：
  - `src/app/api/video-task/create/route.ts:49-66`
- 积分不足时：直接拒绝创建任务（不做“只处理部分视频时长”的半成品交付）。

### 1.4 “订阅权益/优先级”当前怎么判（存在明显偏差）

- 任务优先级目前用“最近一条已支付订单”的 `paymentInterval`（month/year）判断，并通过 `isValid` 做了粗粒度校验（基于 `order.createdAt + creditsValidDays`，仍不是按 `subscription.currentPeriodEnd`）：
  - `src/shared/models/order.ts:375-418`
  - `src/app/api/video-task/create/route.ts:97-105`
- 结论：要做成可审计的权益体系，仍需要基于 subscription 真实账期做判定（见“阶段 B”）。

### 1.5 前端也在算积分（且不一致）

- 前端在多个地方自行计算“每分钟几积分”，部分读取配置，部分仍硬编码 2：
  - `src/shared/blocks/video-convert/project-add-convert-modal.tsx:177-200`（计算成本；历史上曾硬编码 `/2`，已改为按 pointsPerMinute）
  - `src/shared/blocks/video-convert/project-create-flow.tsx:162-165`（成本=duration\*pointsPerMinute，来自 `/api/video-task/getconfig`）
  - `src/shared/blocks/video-convert/convert-add-modal.tsx:112-118`（旧组件/示例，硬编码 2）

### 1.6 存在“客户端可篡改扣费”的点（必须修）

- 合成最终视频接口存在客户端传入 `payCredit` 的“超次付费合成”路径（规则已废弃，需移除，避免客户端篡改/绕过）：
  - `src/app/api/video-task/generate-video/route.ts:25-88`

### 1.7 存在“积分系统安全漏洞”（定价/策略会被直接绕过）

以下接口只要登录即可滥用（或甚至无需登录），会导致任意加积分/刷积分：

- 后台加积分接口必须校验 admin 权限（已补 `PERMISSIONS.CREDITS_WRITE`，仍需补齐审计/幂等）：
  - `src/app/api/credit/add/route.ts:7-52`
- 任意登录用户可删除任意 credit 记录（只要知道 id）：
  - `src/app/api/credit/delete/route.ts:6-34`
- 任何“注册即赠送积分”的接口/前端触发逻辑都必须移除（已取消新用户赠送；否则会被刷爆）。
- 测试接口（consume/refund）必须删除/下线：
  - `src/app/api/payment/credits-test/route.ts`

> 结论：在修补上述安全/一致性问题之前，讨论“定价策略”没有意义，因为可以被直接绕过。

## 2. 与本文件现有“原则/策略”的主要冲突

> 注：此处用于记录“改造前的差距清单”。若代码已按方案落地，应在此处标注“已对齐”或直接删除对应条目，避免误导。

- 文档口径：充值积分按“批次”有效期 12 个月；若 pricing.json 仍存在 `valid_days=60/90/120` 等遗留，需要统一为 12 个月口径。
- 文档口径：字幕段重翻译 / 音频段重生成要扣积分；当前相关接口未扣或可被客户端参数绕过（见 1.6）。
- 文档口径：编辑后重新合成导出 **不扣积分**；当前接口存在 `payCredit` 与“超次付费合成”等逻辑（需要移除，避免规则漂移）。

## 3. 改造原则（KISS + 可审计 + 不信任客户端）

- 单一真相：金额/积分/有效期/扣费规则必须由服务端决定；前端只展示。
- 所有扣费点必须服务端计算，不接受客户端传入金额/积分参数作为权威来源。
- 权益判定基于有效订阅（`subscription.currentPeriodEnd`），不要用“最近订单”。
- 先堵住“刷积分/篡改扣费”的洞，再谈定价和增长。

## 4. 改造方案（建议分阶段上线）

### 阶段 A：先修安全与一致性（必须优先做）

3. 统一 points-per-minute 的回退值

- `src/app/api/video-task/create/route.ts`：默认回退从 2 改为 3（若策略定稿为 3）。

4. one-time（充值积分）有效期统一为 12 个月（按批次到期）

- `src/config/locale/messages/zh/pricing.json` / `src/config/locale/messages/en/pricing.json`：所有 `interval=one-time` 的充值套餐 `valid_days` 统一为 365（口径=12个月），并同步更新“有效期XX天”的展示文案。
- `src/app/api/payment/checkout/route.ts`：继续把 `pricingItem.valid_days` 写入 `order.creditsValidDays`（用于审计/展示）。
- `src/shared/models/credit.ts:calculateCreditExpirationTime()`：one-time 充值积分的 `expiresAt` 按“充值成功起 12 个月”计算（按月对日，无该日期取月末），不要用简单的 `+365 days`，避免闰年/月末导致的到期日偏移。

### 阶段 B：把“扣费规则/订阅状态/积分账本”服务端化（KISS，强一致）

新增一个服务端“计费状态”查询（命名建议：`getUserBillingState(userId)`）统一返回：

- `pointsPerMinute = 3`
- `costs`：
  - `subtitle_retranslate = 1`
  - `audio_regen = 2`
  - `video_translate = 3/min`
- `subscription`：`isActive`、`currentPeriodStart`、`currentPeriodEnd`、`interval`
- `credits`：`subscriptionRemaining`、`topupRemaining`、`totalRemaining`
- `scheduling`：仅供后端调度使用（前端不展示具体数字权益）

约束（已确认）：

- 视频可处理时长只由积分余额决定（不再设置“单视频最长XX分钟”这种独立硬限制）。
- 扣费必须在服务端计算与执行；前端只展示“将消耗多少积分”。
- 订阅积分：按订阅月发放、按订阅月到期（不结转）。
- 充值积分：按“批次”发放，每批次有效期 12 个月，到期批次剩余积分失效。
- 扣减顺序：谁先到期先用谁；同类内也按到期时间先到期先用。

用途：

- `video-task/create` / `generate-subtitle-voice` / `generate-video` 等所有扣费点统一走同一套服务端规则。
- 前端展示余额时，区分展示“订阅积分/充值积分/总积分”，但不展示并发/队列等数字权益。

同时修掉当前“用最近订单判订阅有效性”的做法：

- 以 `getCurrentSubscription()` + `subscription.currentPeriodEnd` 判定是否有效订阅
- interval（月/年）不应影响扣费口径，只影响账期与积分发放节奏

必须修复的积分底座问题（否则“万无一失”做不到）：

- `consumeCredits()` 必须做到两件事，否则后续“按月发放/清零”会更容易踩坑：
  - 扣减顺序正确：严格按到期时间扣（`expiresAt` 越早越先扣，`expiresAt=NULL` 永久有效最后扣）。
  - 扣减算法不能用“offset 分页 + 同时更新 remainingCredits”这种会跳行的方式（会出现“明明够扣却被误判不足”的情况）。
- 所有涉及 credits 增减的 API 必须做权限/幂等/不可篡改（尤其是 admin 赠送/删除、注册赠送、编辑后合成导出扣费）。

### 阶段 B2：订阅积分按月发放（包含年付）

已确认：年付积分按月发放（不是一次性发全年）。

需要注意：当前实现的“订阅赠送积分”是绑定支付回调/renewal 事件的——年付在支付平台侧一年只 renew 一次，因此**仅靠支付平台事件无法做到“每月发放”**。

补充口径：订阅积分按“订阅账期月”（对日顺延）清零/结算，不按自然月（1 号到月底）。

落地方案（已定稿，KISS + 可审计 + 不依赖年付 renewal）：

1. Webhook/支付回调只负责“同步订阅状态”

- 只更新 `subscription.status/currentPeriodStart/currentPeriodEnd/interval/...`，把支付平台作为“扣款周期”的 source of truth。
- 不再试图用“年付 renewal”去驱动每月发放（它一年才来一次）。

2. 内部做一个“订阅月积分发放器”（idempotent）

- Java 核心函数：`ensureSubscriptionMonthlyCredits(subscription)`（或按 `userId` 查最新有效订阅后发放）。
- 输入：用户当前有效 subscription（`ACTIVE/TRIALING/PENDING_CANCEL` 且 `now < currentPeriodEnd`）。
- 计算：用 `subscription.currentPeriodStart` 作为账期锚点，按文档“对日规则（无该日期取月末）”切分出“当前订阅月”的 `periodStart/periodEnd`。
- 发放：若该 `periodStart` 的订阅月 grant 不存在，则创建一条 `credit(grant)`：
  - `transactionScene=subscription`
  - `subscriptionNo=...`
  - `credits=subscription.creditsAmount`（这里必须是“每月订阅积分”，不是全年总额）
  - `expiresAt = periodEnd`（当订阅月期末过期，不结转）
- 幂等：用“订阅号 + periodStart”做幂等键，直接复用现有 `credit.transactionNo` 唯一约束，例如：
  - `transactionNo = sub_month:${subscriptionNo}:${periodStart.toISOString()}`
  - 重试/并发下插入若冲突则视为已发放（查询并返回即可）。

3. 触发机制（Java 定时任务，KISS）

- 定时任务（cron/worker）：周期性扫描所有有效订阅，调用 `ensureSubscriptionMonthlyCredits()` 补齐当前订阅月积分（防止用户不登录也不到账、也方便前台展示余额）。
- 可选增强（非必须，按规模再做）：订阅状态更新（active/renewal）发生时，额外触发一次该函数（避免“刚订阅立刻用却还没扫到”的体验问题）。

4. 与现有发放逻辑的关系（必须改，不然会双发/错发）

- Web 端支付回调/renewal **不再直接发放订阅积分**（避免年付一次性发全年）；只同步订阅状态即可。
- Java 端统一用 `ensureSubscriptionMonthlyCredits()` 发放“当期订阅月”的订阅积分（幂等插入，重复触发安全）。

### 阶段 C：按定价策略精简产品档位并更新 pricing.json（短期可落地）

> 仍使用当前“i18n JSON 作为定价配置”的方式，但把结构收敛到最少可卖的 SKU，减少维护成本与出错面。

建议 SKU（按当前口径，可直接落地）：

- 订阅：标准/高级（按月 + 按年；年付“两个月免费”则年价=月价\*10）

- 充值：60/200/500 积分（建议 valid_days=365，口径=12个月有效期；并按批次独立到期）
- 企业：只展示，不走 checkout

展示口径（面向用户）：

- 不展示“并发/队列/时长上限”等数字权益；只展示价格、周期订阅积分、充值积分与有效期口径。
- “视频可处理时长”不在价格表里用分钟换算展示（避免与扣费口径/实际行为漂移），只强调“按分钟扣积分”。

需要同步修改两份文件（风险：中英文漂移）：

- `src/config/locale/messages/en/pricing.json`

### 阶段 D：中期优化（把业务数值从 i18n 拆出来）

收益：

## 5. 待确认问题（必须定稿，否则无法落地）

### 5.1 已确认

- 买积分/充值不影响用户等级（tier），只是在用户当前基础上增加积分余额。
- 直接影响：视频可处理时长严格由积分余额决定；订阅与充值的差异在于订阅积分（账期月清零/自动发放）与服务体验（后端自动调度，不用数字承诺）。
- 订阅周期边界：优先以 PayPal/Stripe API 返回的开始/结束/下次扣款时间为准；若缺失，再按「对日规则」兜底计算。（见“订阅周期与积分规则”）
- 订阅积分（按月发放/按月过期）：
  - 月付：每个订阅月发放一次；当期内有效，到期失效，不结转。
  - 年付：年付有效期内仍按“每个订阅月”发放一次；当期内有效，到期失效，不结转。
- 充值积分（Top-up，按批次到期）：
  - 每次充值生成一个独立批次；每批次有效期 12 个月，到期批次剩余积分失效。
  - 扣减顺序：谁先到期先用谁；充值批次内部“先到期先用”。
- 年付折扣口径：坚持“两个月免费”（年价 = 月价 \* 10，等效折扣 16.67%），不使用“15% off”口径。
- guest 体系已移除，不再支持访客账号/访客试用（已移除相关接口、组件、文案与配置；后续只需清理遗留数据/配置项）。
- 扣费口径固定为：
  - 视频翻译生成：3 积分/分钟（计费分钟 = `ceil(durationSeconds / 60)` 向上取整，且最少按 1 分钟计费）
  - 字幕段重翻译：1 积分/次
  - 音频段重生成：2 积分/次
  - 编辑后重新合成导出：0（不扣积分；不涉及 GPU 算力，且编辑阶段已扣积分）
- “视频编辑大保存=0积分”：仅保存工程/文案等不触发 GPU 的操作不扣积分；重新合成/导出同样不扣积分（复用已有字幕/音频素材）。
- “限时特惠（$5.99/3 天）”：
  - 计费类型：one-time（一次性购买，无自动续费）。
  - 复购顺延：若在有效期内再次购买，则在**当前到期日**基础上对日对时顺延 3 天；若已到期，则从购买时间起算 3 天。
  - 复购积分生效：若在有效期内再次购买，则**本次新增积分从下一段 3 天周期开始才可用**（不提前透支未来周期）；若已到期后再购，则从购买时间起算立即可用。
  - 多次购买（买了 n 次）：按订单付款时间依次生效，**每段 3 天周期只会生效 1 笔特惠积分批次**，不存在“积分叠加到同一周期一起用”的口径。
  - 权益与“标准版”一致；积分口径与“标准版订阅积分”一致；每次购买产生的积分仅在对应有效期内有效，到期不结转。
- 订阅拒付/未完成/取消：
  - 发起订阅但未完成支付/扣款失败：状态维持订阅前，不生效、不发放订阅积分。
  - 取消订阅：当期权益继续享有至到期日；到期后不自动扣费、不再发放订阅积分，回到普通用户状态。
- 退款：仅支持“未使用的充值积分费用”退款；退款后回收对应充值积分；订阅费用不退款。
- 对日规则（账期口径）：同一日期同一时间扣费；若下月/次年无该日期，则为该月最后一天同一时间。
- 订阅升级/降级（标准↔高级）：下一账期月生效。
- 免费与赠送：取消免费权益；取消新用户一次性赠送；增加“限时特惠”。
- 年付“按月发放订阅积分”触发机制（落地口径）：Webhook 只同步订阅状态；由 **Java 定时任务** 直接扫描 `subscription` 并写入 `credit` 来完成“当期订阅月”积分发放（按对日规则切分账期月），幂等依赖 `credit.transaction_no` 唯一约束（不做历史补发）。
- 积分不足：所有需要扣积分的操作（创建任务/字幕段重翻译/音频段重生成）**直接拒绝**，不允许“只处理部分时长”的半成品交付。
- 支付平台“争议款/拒付（chargeback）”的极端场景：暂不纳入本次改造范围（先跑通主链路，后续遇到再补齐风控与财务口径）。
- 存量数据/灰度切换：无存量数据（未发布），可直接全量按新口径实现（无需迁移/兼容）。
- 多订阅并存策略：V1 口径（KISS）不支持同一用户同时存在多条有效订阅；有效期内再次下单统一引导到 Billing Portal 管理（避免双订阅导致权益重复、对账困难）。
- 后端调度优先级口径（落地到 Java 端）：采用“分层队列 + 加权公平 + 防饿死兜底（aging）+ 并发配额”的主流方案；对外只承诺“更快/最快”，不承诺任何数字（详见「6. 后端调度（Java）实现规范」）。

### 5.2 保留问题（先记录风险，后续再定）

- 「扣费口径严格按文档，且不信任客户端」在“视频时长”上的落地：
  - 现状：创建任务时 `videoDurationSeconds` 来自前端上传参数，存在被篡改从而少扣积分的风险（不满足“万无一失”）。
  - 建议口径：服务端/Java 端以“实际解析的视频时长”为准（例如 ffprobe/媒体元数据），前端时长仅用于展示；扣费使用后端计算结果。
  - 当前处理：先保留（暂不纳入本次改造范围），仅作为风控/反作弊风险项记录；后续若出现争议/被刷，再单独立项补齐。

## 6. 后端调度（Java）实现规范（定稿）

> 适用范围：Java 任务执行端（worker/dispatcher），负责从 `vt_task_main` 拉取待处理任务并执行各步骤。
>
> 目标：付费用户“明显更快”，但免费用户不被饿死；后端更稳、更可控；实现可解释、可审计；避免引入复杂订阅规则到 Java 端（权益判定在 Web 端入队时已固化）。

### 6.1 核心原则（KISS）

- **只做分层，不做复杂打分**：不要把几十条规则塞进一个 priority 公式里。
- **权益判定在入队时固化**：任务入队时已把“是否订阅/是否限时特惠”转成 `priority`（Java 端不再实时查 subscription/credits）。
- **对外不承诺数字**：前端展示“更快/最快”；Java 内部可以用配置参数控制权重、兜底阈值等。
- **永不饿死**：高峰期也必须保证免费队列持续出队（否则会积压爆炸，最终拖垮系统与口碑）。

### 6.2 数据口径（Java 只依赖这些字段）

表：`vt_task_main`

- `status`：待处理任务筛选用（建议仅消费 `pending`）。
- `priority`：任务层级（数值越小越优先；仅用于“入队时固化的层级”）。
- `created_at`：同层级内公平性（越早入队越先处理）。
- `user_id`：用于并发配额（防止单个用户占满资源）。

索引：已存在 `idx_vt_task_main_schedule(status, priority, created_at)`，按本方案的查询顺序命中。

### 6.3 层级定义（Tier → priority 映射）

> 说明：Java 端只按 priority 分层，不关心“月/年”。“包月=包年”“限时特惠=标准”在映射上天然成立。

- `priority=1`：企业（Enterprise，若暂未开放可不产生该优先级任务）
- `priority=2`：付费（Paid：标准/高级/限时特惠）
- `priority=4`：免费（Free：非订阅且无特惠；包括“只充值积分”的用户仍按免费层级处理，避免稀释订阅价值）

> 若未来必须把“仅充值积分”也加速：建议新增 `priority=3`（Top-up），仍保持三层结构，不要把订阅体系打散。

### 6.4 调度算法（主流最佳实践，最小可落地）

采用：**分层队列 + 加权公平（Weighted Round Robin）+ 防饿死兜底（Aging）**。

#### 6.4.1 加权公平（WRR）

逻辑（概念）：worker 在一个调度周期内，按权重比例从不同层级取任务，而不是“永远只取最高优先级”。

推荐实现（KISS）：

- 维护一个可配置的取任务序列（例如：`[paid, paid, paid, paid, free]`），循环执行。
- 每次从对应层级队列取 1 个任务（取不到则跳过尝试下一个层级）。

配置建议（仅内部参数，不对外展示）：

- `schedule.wrr.pattern`：字符串/数组（例如 `paid,paid,paid,paid,free`）
- 或者拆成权重：`schedule.weight.paid` / `schedule.weight.free`

#### 6.4.2 防饿死兜底（Aging）

仅靠 WRR 在极端高峰仍可能导致免费等待过长，因此增加一个**硬兜底**：

- 若存在等待超过阈值的免费任务，则下一次调度**强制优先**取该任务（即使当前轮到 paid）。

配置建议：

- `schedule.free.force_pick_after_seconds`：免费任务最大等待阈值（仅内部使用）

实现要点（避免复杂 SQL）：

- Java 每轮循环额外做一次“是否存在超时免费任务”的轻量查询（只查 1 条）。
- 一旦命中，直接 claim 该任务并执行。

### 6.5 并发配额（防止单用户占满资源）

> 这不是对外卖点，是系统稳定性底线。

建议采用“每用户并发上限”，按层级设置不同上限：

- `schedule.user_concurrency_limit.paid`
- `schedule.user_concurrency_limit.free`

实现（KISS）：

> 状态枚举（已确认）：`pending / processing / completed / failed / cancelled`

1. Java 选中候选任务后，在**同一个 DB 事务内**查询该 `user_id` 当前处于“执行中”状态的任务数：**只统计 `status='processing'`**。
2. 超过上限则跳过该任务（不改状态），继续选下一个候选。

> 如需更强一致性（避免并发竞态超限），再引入“用户令牌表/分布式锁”；V1 可以先不做，保持简单。

### 6.6 任务 claim（必须保证不重复消费）

Java 端从 DB 取任务必须是“原子 claim”，避免多个 worker 抢到同一条任务。

推荐做法（PostgreSQL）：

1. 在事务内 select 候选任务并锁住：
   - `... where status='pending' and priority=? order by created_at asc for update skip locked limit 1`
2. 立即 update 该任务为执行中状态，并写入 `started_at`：
   - `update vt_task_main set status='processing', started_at=now(), updated_at=now() where id=? and status='pending'`

更推荐的“一条 SQL 原子 claim”（Java 端可直接照抄）：

```sql
WITH picked AS (
  SELECT id
  FROM vt_task_main
  WHERE del_status = 0
    AND status = 'pending'
    AND priority = ?
  ORDER BY created_at ASC
  FOR UPDATE SKIP LOCKED
  LIMIT 1
)
UPDATE vt_task_main
SET status = 'processing',
    started_at = NOW(),
    updated_at = NOW()
WHERE id IN (SELECT id FROM picked)
  AND status = 'pending'
RETURNING *;
```

注意：

- **必须**带 `status='pending'` 条件更新，更新行数=1 才算 claim 成功。
- 失败则说明被别的 worker 抢走，直接重试下一轮。
- 任务状态机（已确认）下，Java 端只需要做：`pending -> processing -> (completed|failed|cancelled)`，不要新增 `running/retrying` 等状态，避免口径漂移。

### 6.7 对现有 Web 端的要求（契约）

Web 端（Next.js）在创建任务时需：

- 先按文档口径扣积分，扣费成功才入队（已实现）。
- 在入队时写入 `vt_task_main.priority`：
  - 有效订阅或有效限时特惠 → `priority=2`
  - 否则 → `priority=4`

Java 端不得：

- 再次根据 subscription/order/credits 去“重算”priority（避免口径漂移与竞态）。

### 6.8 监控与压测（必须有，否则调度等于瞎跑）

至少要有这些指标（内部用）：

- 各层级 pending 队列长度（paid/free/enterprise）
- 各层级任务等待时长分布（p50/p90/p99）
- 强制兜底触发次数（free aging hit count）
- 每用户并发拒绝次数（skipped due to concurrency cap）

## 7. 回归/验收清单（建议）

- 充值/订阅支付成功：订单入库、积分发放正确、expiresAt 逻辑正确（充值=12个月；订阅=订阅月期末到期；限时特惠=权益到期）。
- 订阅续费/账期滚动：续费订单生成、订阅状态同步、进入新订阅月后可自动发放当期订阅积分（不依赖年付 renewal 事件）。
- video-task/create：服务端按 points-per-minute 计费；积分不足直接拒绝（不做“部分时长”）。
- generate-video：扣费不可被客户端篡改；每日次数限制/付费规则符合预期。
- 管理端：加/删积分接口只有 admin 可用；普通用户无法绕过。
