# 时间线字幕切割功能 Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 在视频编辑器中实现“按当前播放头切割字幕片段”的完整能力：只拆分时间轴和字幕行，不自动裁剪或生成音频；切割后文本先保持不变，由用户手动调整文本，并分别点击“重新配音”为两段生成各自语音，同时保证刷新恢复、再次拖拽、再次保存、重新合成、历史数据兼容和整体性能都正确。

**Architecture:** 采用“前端即时响应 + 后端原子落库 + 显式状态字段恢复”的纯字幕切割方案。Next API 只负责把 `translate_srt` 与 `gen_srt` 两份 JSON 数组同步拆分并写回 DB，不调用外部 Python/VAP 做任何音频裁剪。切割后两条新译文片段进入“待重新配音”状态；原声试听对 split 新行直接使用现有 `vocalAudioUrl + 时间窗` 模式，不等待也不生成新的 source 音频文件。

**Tech Stack:** Next.js App Router, React 19, TypeScript, next-intl, Drizzle ORM / PostgreSQL JSONB, Vitest.

## Chunk 1: 产品定义与完整范围

### 1.1 本次采用的产品定义

这次明确按用户需求落地，**切割不是音频处理功能，而是字幕结构编辑功能**：

- 切割只改变：
  - 时间轴片段结构
  - 字幕工作台行结构
  - `translate_srt` / `gen_srt` 的时间与行数
- 切割不会自动：
  - 裁剪原始音频
  - 复制旧译音频
  - 生成新译音频
  - 生成新 source 分段音频文件
- 切割后：
  - 两个新片段的文本初始都保持与原片段相同
  - 用户手动改文本
  - 用户分别点击“重新配音”生成各自语音

### 1.2 这次明确纳入范围

- 在 `TimelinePanel` 工具栏和快捷键 `S` 上触发切割。
- 切割前强制 flush：
  - `pendingTimingMap`
  - 当前行未落库文本草稿
- `translate_srt` 与 `gen_srt` 必须在一个 DB 事务里同步拆分并写回。
- split 后两个新译文子片段都进入“待重新配音”状态。
- 原声试听对 split 新行必须直接走 `vocalAudioUrl + 时间窗`，不能先请求不存在的 `split_audio/audio/{id}.wav` 再等 4 秒 fallback。
- `update-subtitle-item`、`update-subtitle-timings`、`generate-video`、工作台 pending、刷新恢复都必须兼容 split 后的新数据。

### 1.3 明确不纳入范围

- 不做撤销 / 重做。
- 不做批量多选切割。
- 不做自动裁剪原始音频。
- 不做自动继承旧译音频。
- 不做外部 VAP / Python 改造。

### 1.4 为什么这版不需要 VAP

- 切割动作本身只是字幕数据重排，不需要媒体处理。
- 用户已经明确：切割后文本保持不变，后续由用户手动调整文本并分别重新配音。
- 因此 VAP 不再是 split 的前置依赖。
- 当前仓库只需要处理两类后续音频语义：
  - **译文音频：** split 后清空，等待用户后续重新生成。
  - **原声试听：** split 新行不走分段文件，直接走 `vocalAudioUrl + 时间窗 fallback`。

### 1.5 完整性标准

要算“完整实现”，必须同时满足：

1. 时间轴翻译轨从 1 条变 2 条。
2. 原文参考轨也从 1 条变 2 条，保持工作台索引对齐。
3. 工作台从 1 行变 2 行。
4. 两条新译文都被识别为“待重新配音”。
5. 刷新后结构不丢、不误判历史数据。
6. split 后再次拖拽、再次保存、再次重新配音、再次合成都可继续工作。
7. 不引入整页重载、重复音频请求、明显主线程卡顿。

## Chunk 2: 数据契约与兼容规则

### 2.1 主键与顺序契约

1. **唯一主键统一为 `id`。**
2. `seq` 只表示显示/导出顺序，切割后统一重排为 `1..n`。
3. split 后生成的新 `id` 必须继续兼容 `update-subtitle-timings` 的时间编码规则。
4. 新 `id` 生成规则继续采用“纯数字前缀扩展”，禁止 `0001a` 这种会破坏当前正则的做法。

示例：

```text
原始：0001_00-00-00-000_00-00-04-000

第一次切割：
- 左：00010001_00-00-00-000_00-00-02-000
- 右：00010002_00-00-02-000_00-00-04-000
```

### 2.2 `translate_srt` 完整字段契约

#### 必须保留/重建

- `id`
- `seq`
- `start`
- `end`
- `txt`
- `timing_rev_ms`

#### 必须清空

- `audio_url`
- `audio_rev_ms`
- `vap_draft_audio_path`
- `vap_draft_duration`
- `vap_tts_job_id`
- `vap_tts_request_key`
- `vap_tts_updated_at_ms`
- `vap_tr_job_id`
- `vap_tr_request_key`
- `vap_tr_updated_at_ms`

#### 必须新增/写入

- `vap_voice_status: 'missing'`
- `vap_needs_tts: true`
- `vap_split_parent_id: <oldId>`
- `vap_split_operation_id: <uuid>`
- `vap_draft_txt: <effective current text>`

### 2.3 `gen_srt` 完整字段契约

#### 必须保留/重建

- `id`
- `seq`
- `start`
- `end`
- `txt`

#### 不做的事

- 不生成新的 `split_audio/audio/{newSourceId}.wav`
- 不要求 `audio_url` 指向新的 source 文件

#### 必须新增/写入

- `vap_source_mode: 'fallback_vocal'`
- `vap_source_segment_missing: true`
- `vap_source_split_parent_id: <oldSourceId>`
- `vap_split_operation_id: <uuid>`

### 2.4 历史数据兼容规则

**绝不能直接把 `audio_url === ''` 当成缺音频。**

兼容判定顺序固定如下：

```text
translate_srt:
1. 若存在 vap_voice_status，则按显式状态处理
2. 否则：vap_draft_audio_path > audio_url > adj_audio_time/{id}.wav（历史兼容）

gen_srt:
1. 若存在 vap_source_mode，则按显式模式处理
2. 否则：audio_url > split_audio/audio/{id}.wav > vocalAudioUrl fallback
```

### 2.5 运行中操作冲突规则

目标片段在以下任意状态下，**禁止 split**：

- `vap_tr_job_id` 非空
- `vap_tts_job_id` 非空
- 当前行正在“应用配音保存”
- 当前行存在未 flush 的 timing/text 改动

## Chunk 5: 关键实现细节说明

### 5.1 为什么 split 后不保留旧译音频

- 旧译音频对应的是切割前的完整文本和完整时间窗
- split 后直接继承会出现：
  - 内容语义不对应
  - 时长超窗
  - 用户误以为可直接使用
- 所以必须清空，等待用户手动调整文本并分别重新配音

### 5.2 为什么 `gen_srt` 也要拆，但不切 source 音频

- 工作台当前是按 `srt_source_arr[i]` 与 `srt_convert_arr[i]` 索引对齐构建的
- 如果只拆 `translate_srt` 不拆 `gen_srt`，工作台与原文参考轨会错位
- 但“拆 source 行”和“切 source 音频文件”不是一回事
- 本需求只要求前者，不要求后者

### 5.3 为什么需要显式状态字段

- 历史数据大量依赖路径推导
- 若不引入：
  - split 新行会被误当成有音频
  - 历史行会被误当成缺音频
- 因此至少需要：
  - `vap_voice_status`
  - `vap_needs_tts`
  - `vap_source_mode`
  - `vap_source_segment_missing`

### 5.4 为什么完整方案仍然要改服务端 guard

当前 `generate-video` 只看 `audio_rev_ms / timing_rev_ms`，并不校验音频是否真的存在。  
所以即使前端拦住，服务端也必须对 `vap_needs_tts` / `vap_voice_status` 做 guard，避免脏合成。
